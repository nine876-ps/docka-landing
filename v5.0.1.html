What is a Runbook

A runbook verifies that an agent can complete a task — in one shot, with minimum tokens.

A runbook is a machine-readable definition of one job your users need to complete — with a measurable end state, not a list of steps. Preconditions, API calls, expected responses, and a binary exit criterion: did the state change or not?

The key part: we run it. Against a real agent. In a blank sandbox, every time. We don't ask the model if it succeeded — we query the API or database and check the actual state. One shot. Graded by outcome.

create-db-with-schema.yaml
# task: "Create a database, design a schema from this # description, and connect it to my backend" # user story: spin up a DB from natural language one_shot: true # must complete without human input max_tokens: 4000 # target ceiling per run steps: - id: create-database action: POST {{BASE_URL}}/databases body: {name: "orders", region: "us-east-1"} extract: {db_id: .id, host: .host} expect: {status: 201} - id: infer-schema action: POST {{BASE_URL}}/databases/{{db_id}}/schema/infer body: {description: "orders, line items, users, payments"} extract: {schema_id: .schema.id} expect: {body.schema.tables: "> 0"} - id: connect-backend action: POST {{BACKEND_URL}}/connections body: {host: "{{host}}", schema_id: "{{schema_id}}"} expect: {body.status: "connected"} exit_criteria: success: connect-backend.body.status == "connected" one_shot_pass: no human input required retry: [429, 503] fail: [401, 500]
One-Shot Target

The task must complete without asking for clarification or human intervention. No loops, no retries on ambiguity.

Execution-Graded

Success is measured by API or database state — not what the model says it did. Binary: the state changed, or it didn't.

Continuously Re-measured

When your docs change or a new model ships, we re-run. Regressions surface before users hit them.

A runbook that passes is a use-case your users' agents can one-shot. A failing runbook is revenue at risk.

API Coverage

Do your APIs expose everything an agent needs to complete the use-case — including preconditions, intermediate states, and cleanup paths? We run baseline use-cases and record exactly which endpoints agents hit, in what order, and which calls return unexpected responses or gaps.

Authentication Flows

Can an agent authenticate and maintain a valid session for the full duration of a use-case — without asking a human for credentials or re-auth mid-task? We measure whether auth succeeds in one shot and whether the agent recovers when tokens expire during a long-running task.

Error Handling

Are your error codes specific enough for an agent to decide what to do next — retry, abort, or escalate? We deliberately trigger error paths in baseline runs and observe whether the agent recovers correctly or loops trying variations of the same failing call.

Rate Limiting

Is your backoff behavior documented and predictable enough for an agent to handle without human oversight? We observe retry patterns and token cost spikes in baseline runs — unexplained cost spikes often trace back to unhandled 429s with no documented backoff strategy.

Runbook Completeness

Does a machine-readable runbook exist for this use-case — with preconditions, step sequence, expected API responses, and a binary exit criterion? This is the core intervention: we measure baseline success, write the runbook, and re-measure. The delta is the value.

State Management

Are preconditions documented and side effects predictable — so the agent knows what state it needs to start from and what state it needs to produce? We verify end state after every run via API or database query. The success criterion is always a measurable state change, never model opinion.

Context Quality

Is the right information reachable for agents in minimal fetches — without pulling irrelevant docs or navigating your documentation tree from scratch? Token cost per use-case is a direct proxy: bloated token counts mean agents are fetching too much to find what they need.

Observability

Can you tell — right now — which of your use-cases are failing for agents, and which specific doc pages are causing the breaks? The Runbook Map is the observability layer: every use-case has a readiness level, a linked doc page, and a re-run cadence. When your API changes, we re-measure automatically.

FAQ

Questions we get on every call.

How do you actually measure this?
We spin up a sandbox that reproduces your product, run a real AI agent on raw docs only — it doesn't know it's being tested — then check the actual end state via API or database. Did the rows change? Was the user migrated? Is the index built? That's the grade. We run each use-case 3–5 times per model (Claude, GPT, Gemini) to catch variance, then run again with the runbook in context for the A/B delta.
What exactly counts as "success"?
Success means the measurable end state of the task was reached — not that the model said it succeeded. We define this per use-case before the run: e.g., "users and grants in old system match users and grants in new system." We query the API or database after the run and compare. Binary: pass or fail-path. No partial credit, no model opinion.
What are the steps and how long does it take?
We collect your use-cases (you send them, or we extract from docs + GitHub issues) → run baseline × 3 models → assign readiness levels → build the map → prioritize by importance × gap → write runbooks → publish → re-measure. A Scan (1–2 use-cases) takes ~1 week. A full Audit (your complete JTBD list) takes ~3 weeks.
How is this different from MCP servers or agent skills?
MCP servers and skills need to be discovered and installed. In practice, ~90% of users never install them — they just use the agent with whatever context is already available. Runbooks are agent-agnostic and zero-install: published via content-negotiation and llms.txt so agents pick them up automatically from your existing docs. No setup, no SDK, nothing to deploy.
I already have an llms.txt. Does that count?
An llms.txt is a good start, but it's only read when an agent is explicitly prompted to check it. We optimize what the agent pulls itself during a task — the docs it actually fetches, the API it actually calls — and fix those to be machine-actionable. Even teams who wrote their own MCP + skills found that once there's no install required, the value becomes clear.
What if some of my use-cases already work fine?
We report it honestly and leave those use-cases alone. If an agent already one-shots a task on raw docs, there's no runbook to write — and you don't pay for work that isn't needed. That's a feature of how we prioritize: importance × gap. Low gap means no intervention. High gap on a high-priority use-case means we go there first.
