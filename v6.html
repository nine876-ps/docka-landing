The problem

Agent tasks fail — or burn 100× the tokens they should.

When a user tries to complete a task through an agent, two things go wrong: the task doesn't finish at all, or it burns hundreds of thousands of tokens doing what should take hundreds. As more of your users reach for agents, this scales with them — one broken flow × your entire user base × every session.

The Problem

Agents fail in production. Silently. Repeatedly.

Without a runbook, one use-case can spike to 376k tokens. With one: 21k.

When API docs don't match agent expectations, models loop: follow the spec, hit an error, retry with variations, exhaust context. Measured on NocoDB — 7 of 15 use-cases failed on raw docs, token consumption from 57k to 376k. With runbooks: 0 of 15 failed, all within 21k–29k tokens.

You won't know it's broken until a user reports it. If they ever will.

Execution-graded failure is binary and silent — the API returns a state, or it doesn't. There's no crash log. Agents stall at the same edge cases across sessions, and users churn without a word.

What is a Runbook

A machine-readable definition of one job-to-be-done.

01

One task. One measurable end state.

Each runbook covers exactly one use-case — with preconditions, API calls, expected responses, and a binary exit criterion: did the state change or not?

02

Written for agents, not humans

Explicit auth steps, exact API calls, enum values, error patterns, and a machine-verifiable success signal. Not a tutorial — an execution spec.

03

Graded by real execution, not model opinion

We query the API or database after the run and check the actual state. Binary: pass or fail. No partial credit, no model opinion.

04

Continuously re-measured

When your docs change or a new model ships, we re-run. Regressions surface before users hit them.

create-db-with-schema.yaml
# task: "Create a database, design a schema from this # description, and connect it to my backend" one_shot: true # must complete without human input max_tokens: 4000 # target ceiling per run steps: - id: create-database action: POST {{BASE_URL}}/databases body: {name: "orders", region: "us-east-1"} extract: {db_id: .id, host: .host} expect: {status: 201} - id: infer-schema action: POST {{BASE_URL}}/databases/{{db_id}}/schema/infer body: {description: "orders, line items, users, payments"} extract: {schema_id: .schema.id} expect: {body.schema.tables: "> 0"} - id: connect-backend action: POST {{BACKEND_URL}}/connections body: {host: "{{host}}", schema_id: "{{schema_id}}"} expect: {body.status: "connected"} exit_criteria: success: connect-backend.body.status == "connected" one_shot_pass: no human input required retry: [429, 503] fail: [401, 500]
One-Shot Target

The task must complete without asking for clarification or human intervention. No loops, no retries on ambiguity.

Execution-Graded

Success is measured by API or database state — not what the model says it did. Binary: the state changed, or it didn't.

Continuously Re-measured

When your docs change or a new model ships, we re-run. Regressions surface before users hit them.

A runbook that passes is a use-case your users' agents can one-shot. A failing runbook is revenue at risk.

What We Measure

Eight dimensions where products fail agents.

Each maps to a specific pattern we've observed: agents looping, burning tokens, or silently producing wrong state.

◫

API Coverage

Do your APIs expose everything an agent needs — including preconditions, intermediate states, and cleanup paths?

🔑

Authentication Flows

Can an agent authenticate and maintain a valid session for the full duration of a use-case without human re-auth?

⚠

Error Handling

Are your error codes specific enough for an agent to decide what to do — retry, abort, or escalate?

⏱

Rate Limiting

Is your backoff behavior documented and predictable enough for an agent to handle without human oversight?

📖

Runbook Completeness

Does a machine-readable runbook exist — with preconditions, step sequence, expected responses, and a binary exit criterion?

⬡

State Management

Are preconditions documented and side effects predictable — so the agent knows what state it needs to produce?

💬

Context Quality

Is the right information reachable in minimal fetches — without pulling irrelevant docs?

◉

Observability

Can you tell — right now — which use-cases are failing for agents, and which docs are causing the breaks?

FAQ

Questions we get on every call.

How do you actually measure this?
We spin up a sandbox that reproduces your product, run a real AI agent on raw docs only — it doesn't know it's being tested — then check the actual end state via API or database. Did the rows change? Was the user migrated? Is the index built? That's the grade. We run each use-case 3–5 times per model (Claude, GPT, Gemini) to catch variance, then run again with the runbook in context for the A/B delta.
What exactly counts as "success"?
Success means the measurable end state of the task was reached — not that the model said it succeeded. We define this per use-case before the run: e.g., "users and grants in old system match users and grants in new system." We query the API or database after the run and compare. Binary: pass or fail. No partial credit, no model opinion.
What are the steps and how long does it take?
We collect your use-cases → run baseline × 3 models → assign readiness levels → build the map → prioritize by importance × gap → write runbooks → publish → re-measure. A Scan (1–2 use-cases) takes ~1 week. A full Audit (your complete JTBD list) takes ~3 weeks.
How is this different from MCP servers or agent skills?
MCP servers and skills need to be discovered and installed. In practice, ~90% of users never install them — they just use the agent with whatever context is already available. Runbooks are agent-agnostic and zero-install: published via content-negotiation and llms.txt so agents pick them up automatically from your existing docs. No setup, no SDK, nothing to deploy.
I already have an llms.txt. Does that count?
An llms.txt is a good start, but it's only read when an agent is explicitly prompted to check it. We optimize what the agent pulls itself during a task — the docs it actually fetches, the API it actually calls — and fix those to be machine-actionable. Even teams who wrote their own MCP + skills found that once there's no install required, the value becomes clear.
What if some of my use-cases already work fine?
We report it honestly and leave those use-cases alone. If an agent already one-shots a task on raw docs, there's no runbook to write — and you don't pay for work that isn't needed. That's a feature of how we prioritize: importance × gap. Low gap means no intervention. High gap on a high-priority use-case means we go there first.
