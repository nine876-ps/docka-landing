THE INVISIBLE BOTTLENECK

The agent is not broken.
Your product environment is.

Teams integrating AI agents with their product often see uneven results. They blame the model, try a different provider, get the same thing. The real problem is usually the product itself — not how it's built, but how it's exposed to agents.

Undocumented API error codes mean the agent retries blindly, spending tokens on guesses instead of advancing the task. Workflows documented for humans — with implicit context, screenshots, and assumed knowledge — mean the agent reverse-engineers your product step by step, then fails at the edge case on step four. API responses that signal failure with a 200 OK mean the agent has no idea it went wrong until three steps later. A UI flow with no machine-readable runbook means every agent session starts from zero.

These are product-readiness problems, not agent problems. And they compound. A product with ambiguous APIs and no structured runbooks will defeat any agent you give it to. A product with clear task definitions, documented failure modes, and verified runbooks will make any agent dramatically more effective — without changing a single line of agent code.

The Problem

Agents fail in production. Silently. Repeatedly.

Agents burn tokens chasing broken docs

When your API docs don't match reality, agents loop: they follow the spec, hit errors, retry with variations, exhaust context, and fail. One failing job can burn 100× the expected token cost.

At scale, 8 of 10 agent sessions will fail

Undocumented edge cases, missing auth flows, and ambiguous error codes turn every agent interaction into a coin flip. You don't know which workflows are broken until a user reports it.

The Fix

One runbook. Every agent. Zero install.

A Runbook is a machine-readable, execution-graded playbook that tells AI agents exactly how to complete a task in your product — step by step, with preconditions, API calls, expected responses, and failure modes.

Unlike documentation written for humans, Runbooks are structured for agents: they minimize ambiguity, handle errors explicitly, and define exit criteria.

runbook.yaml
# runbook: create-search-index.yaml # Task: Create and populate a search collection preconditions: env: [API_KEY, BASE_URL] steps: - id: authenticate action: POST {{BASE_URL}}/auth/token body: {apiKey: "{{API_KEY}}"} extract: {token: .token} expect: {status: 200} - id: create-collection action: POST {{BASE_URL}}/collections headers: {Authorization: "Bearer {{token}}"} body: {name: "products"} extract: {id: .id} expect: {status: 201} on_error: {409: use_existing} - id: bulk-index action: POST {{BASE_URL}}/collections/{{id}}/bulk body: {documents: "{{documents}}"} expect: {status: 200, body.errors: 0} exit_criteria: success: bulk-index.status == 200 retry: [429, 503] fail: [401, 500]

Maturity Levels

Four levels of agent readiness.

Products progress through four levels. Each level represents a qualitative shift in what autonomous agents can accomplish.

1
Unfunctional
2
Semi-functional
3
Fully-functional
4
Optimized

Level 1 — Unfunctional

Agents cannot complete basic tasks.

API calls fail or produce incorrect results. No structured agent guidance exists. Every workflow is a guess.

Signals
  • No runbook
  • Docs out of date
  • No error codes documented
  • Auth undocumented
Capability

Agents fail >90% of attempts.

Level 2 — Semi-functional

Some tasks work, inconsistently.

Authentication exists but error handling is incomplete. Edge cases cause unpredictable failures.

Signals
  • Basic auth documented
  • Partial runbooks
  • No retry logic
  • Missing error codes
Capability

Agents succeed <50% of attempts.

Level 3 — Fully-functional

Core tasks work reliably.

Auth, error handling, and retry logic are fully documented. Agents succeed predictably for known workflows.

Signals
  • Full auth docs
  • Core runbooks
  • Error codes documented
  • Retry guidance
Capability

Agents succeed >80% of attempts.

Level 4 — Optimized

Production-grade agent readiness.

All known agent interactions are covered. Continuous monitoring catches regressions before users do.

Signals
  • Complete runbook coverage
  • CI monitoring
  • Sandbox testing
  • Regression alerts
Capability

Agents succeed >95% of attempts.

How We Score

Consistent evaluations.

Agent Readiness evaluates 60+ criteria using LLMs, which introduces a challenge: non-determinism. We solved this by grounding each evaluation on the previous report. Before the fix, variance averaged 7% with spikes to 14.5%. After grounding, variance dropped to 0.6%.
